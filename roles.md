# Volunteer Roles

AAO Data Lab is entirely volunteer-run. We organize contributors across several disciplines — all of them essential to doing this work well. You can contribute across roles, and your role can evolve over time.

> Not sure where you fit? Read [Mission & Ethos](./mission-ethos.md) to understand what we are building, then come back here.

---

## Software Developer

**What you build:** Web applications, interactive maps, data APIs, and the shared component library.

**Skills:** JavaScript, TypeScript, React, Python, Node.js, SQL, REST APIs, Mapbox/Leaflet

**Good first issues:**
- Frontend component fixes and improvements labeled `good first issue`
- Bug fixes in existing tools
- API endpoint additions

**What "good" looks like:** Clean React components, accessible markup, responsive layouts, and code that someone else can maintain in 6 months.

**Progression:** Contributor → Reviewer → Maintainer

---

## Data Analyst / Data Engineer

**What you build:** Cleaned public datasets, reproducible Jupyter analysis notebooks, data pipelines that ingest GTFS / BTS / Census feeds, and automated economic models.

**Skills:** Python (pandas, geopandas, numpy, scipy), SQL, Jupyter, dbt, Plotly, Altair, QGIS

**Good first issues:**
- Issues labeled `data`
- Documenting existing datasets with source citations and methodology notes
- Data quality audits and anomaly detection

**What "good" looks like:** Notebooks that re-run cleanly top to bottom, clear methodology headers, and outputs that a non-technical person can read.

**Progression:** Contributor → Dataset Owner → Data Lead

---

## UI/UX Designer

**What you build:** Component design specs, user flows, accessibility audits, the AAO Data Lab design system, and visual assets.

**Skills:** Figma, accessibility (WCAG 2.1), design systems, mobile-first layouts, data visualization design, user research

**Good first issues:**
- Issues labeled `design`
- Accessibility improvements (contrast, keyboard nav, screen reader flow)
- Mobile responsiveness fixes
- Improving data visualization clarity

**What "good" looks like:** Clear Figma specs that developers can implement without back-and-forth, and designs that hold up at 375px as well as 1440px.

**Progression:** Contributor → Design Lead

---

## Policy Analyst

**What you build:** Legislative context documents, source credibility reviews, policy landscape summaries, and issue framing guides for data products.

**Skills:** Policy research, legislative tracking, public finance, transportation policy, technical writing

**Good first issues:**
- Issues labeled `research` or `policy`
- Reviewing data visualizations for accuracy and appropriate framing
- Writing background briefs that developers use to scope tools

**What "good" looks like:** Concise, well-sourced documents that give technical contributors the context they need to build accurate tools without getting policy details wrong.

**Progression:** Contributor → Policy Reviewer

---

## Researcher

**What you build:** Literature reviews, comparable program analyses (peer states, international examples), citations and data sourcing, and background research documents.

**Skills:** Academic or professional research, database access, citation management, technical summarization

**Good first issues:**
- Comparing Ohio ridership models to peer state implementations
- Finding and documenting comparable transit investments and their measured outcomes
- Sourcing and documenting public datasets

**What "good" looks like:** Structured research memos with clear source citations, useful to a developer who needs to understand the domain quickly.

---

## Project Manager / Coordinator

**What you build:** Issue scoping documents, user stories, milestone tracking, cross-discipline coordination, and project status summaries.

**Skills:** Project coordination, writing, stakeholder communication, systems thinking, issue triage

**Good first issues:**
- Helping scope and write detailed `good first issue` tickets
- Coordinating between developers and designers on an active feature
- Writing a project brief or scoping document for an upcoming tool

**What "good" looks like:** Issues that are so well-scoped a developer can pick them up and start without asking questions. Milestones that reflect realistic sequencing.

**Progression:** Contributor → Project Coordinator → Program Lead

---

## Communicator / Technical Writer

**What you build:** Documentation, README files, onboarding guides, blog posts, and outreach materials for partners and academic programs.

**Skills:** Technical writing, plain language, Markdown, documentation systems, editing

**Good first issues:**
- Issues labeled `documentation`
- Improving existing docs for clarity or completeness
- Writing user-facing help text for a data tool

**What "good" looks like:** Documentation a first-time contributor can follow without asking for help. No unexplained jargon.

---

## Maintainer

Maintainers are experienced contributors who have demonstrated consistent, high-quality work across multiple cycles. They:

- Triage and label new issues
- Review and merge pull requests with care
- Mentor new contributors through onboarding
- Make architectural and scope decisions in collaboration with other maintainers
- Represent the project in external conversations with partners and institutions

**Maintainer status is invitation-based.** If you have made 5 or more quality contributions and are interested in stepping up, say so in `#dev-general` on Slack. We will have a conversation.

---

## A Note on Cross-Discipline Contribution

The most impactful contributors at AAO Data Lab often work across disciplines. A developer who understands the economic model they are building is more effective than one who does not. A designer who has read the policy brief makes better data visualization choices. A policy analyst who can write a Jupyter cell is extraordinarily valuable.

We encourage everyone to learn adjacent skills over time. Our documentation, public data sources, and Slack community are all here to support that.
